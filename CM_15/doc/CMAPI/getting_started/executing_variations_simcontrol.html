Executing variations using SimControl¶

To start and control an individual simulation process, a cmapi.SimControl object must be used. There are multiple implementations of cmapi.SimControl:

  • cmapi.SimControlInteractive

  • cmapi.SimControlBatch

  • cmapi.SimControlBatchCM4SL

They share a common interface through cmapi.SimControl and exhibit various additional functions depending on the capabilities and configuration of the underlying simulation process

Before the simulation can be started, the newly created SimControl must be set up properly. Initially the prepared variation containing the Test Run parameters must be provided with cmapi.SimControl.set_variation():

simcontrol = cmapi.SimControlInteractive()
simcontrol.set_variation(variation)

Next, all cmapi.Application involved in the simulation must be passed to the SimControl.

A cmapi.CarMaker simulation program is the minimum required application to run a simulation. Out of all additional runtime programs (cmapi.GPUSensor, further CarMaker Applications in a SimNet, co-simulation partners) one single CarMaker application must be designated master, and is supplied with cmapi.SimControl.set_master(). This can either be a cmapi.CarMaker or, for SimControlInteractive, an cmapi.ApoServer object, indicating a connection to an externally started CarMaker simulation program.

GPU sensors required for a simulation may be set by cmapi.SimControl.set_gpusensors(). The list must contain enough cmapi.GPUSensor objects for the configured TestRun. The individual list entries are mapped to the used cmapi.SensorCluster in the order they appear in the variation’s list returned by cmapi.Variation.get_required_sensor_clusters(). Alternatively GPUSensor may be explicitly set for an index with cmapi.SimControl.set_gpusensor().

This results in the lines:

master = cmapi.CarMaker()
gpu_sensors = [cmapi.GPUSensor(), cmapi.GPUSensor()]
await simcontrol.set_master(master)
await simcontrol.set_gpusensors(gpu_sensors)

Note

Manual setup of runtime applications can be circumvented by using a Runtime instead.

SimControl does not automatically start all involved applications when the simulation is started. It’s the user’s responsibility to always start and stop all processes belonging to the runtime environment, with the SimControl only coordinating the participants. An exception to this are processes, which can not be re-used for following simulations and have a defined lifetime which is congruent with the simulation time (e.g., CarMaker process existing while a simulation is performed in cmapi.SimControlBatch mode). These are started automatically alongside cmapi.SimControl.start_sim() and are automatically terminated when the simulation terminates or is stopped by cmapi.SimControl.stop_sim().

Additionally, cmapi.SimControlInteractive requires an APO connection, which needs to be established once by a call to cmapi.SimControlInteractive.start_and_connect() and may be re-used for all following simulations:

await simcontrol.start_and_connect()

Afterwards the simulation can be started with cmapi.SimControl.start_sim():

await simcontrol.start_sim()

The simulation is now running as a background task. The automation script is now suspended until either a user defined event occurs or the simulation has finished. Examples for both are given. But first, the simulation speed is decreased to ease synchronization with the automation script:

simcontrol.set_realtimefactor(1.0)

Using the SimControlInteractive, the APO connection can be used to pass messages between the simulation program and the automation script. As one possible benefit of this, a cmapi.SimCondition can be installed during runtime, which is either evaluated on client side based on subscribed ApoQuantities’ values or on simulation side, based on a real time expression. The functions therefore are cmapi.SimControlInteractive.create_quantity_condition() resulting in a cmapi.QuantityCondition and cmapi.SimControlInteractive.create_rtexpr_condition() resulting in a cmapi.RTExprCondition. Once generated by the SimControl, the automation script can be suspended until the condition evaluates to true, by calling its cmapi.SimCondition.wait() coroutine function.

condition = simcontrol.create_quantity_condition(lambda car_v: car_v > 10.0, "Car.v")
await condition.wait()

At this point, further functions could be called, e.g. to perform a cmapi.SimIO.read_dva() call, using the cmapi.SimIO member of the current SimControl to take some measurements. Once all measurements are taken, the script should be suspended once more, until the simulation ends. This is accomplished, by awaiting a cmapi.SimStateCondition associated with simulation state cmapi.ConditionSimState.finished:

await simcontrol.create_simstate_condition(cmapi.ConditionSimState.finished).wait()

As part of a proper clean-up, the runtime applications and background tasks should be stopped by calling cmapi.SimControlInteractive.stop_and_disconnect() after the last simulation has concluded:

await simcontrol.stop_and_disconnect()

Full example¶

from parametrization_full_example import *


async def main():
    variation, variation_trailer = await make_variations()

    cmapi.logger.info(f"Executing with SimControl...")

    simcontrol = cmapi.SimControlInteractive()
    simcontrol.set_variation(variation)

    master = cmapi.CarMaker()
    await simcontrol.set_master(master)

    gpu_sensor = cmapi.GPUSensorNX()
    await simcontrol.set_gpusensors([gpu_sensor])

    await simcontrol.start_and_connect()
    await gpu_sensor.start()
    await simcontrol.start_sim()

    # Interactive commands
    condition = simcontrol.create_quantity_condition(lambda car_v: car_v > 10.0, "Car.v")
    await condition.wait()

    time_10, = await simcontrol.simio.dva_read_async("Time")
    cmapi.logger.info(f"{variation.get_name()}: Reached speed 10 m/s after {time_10} seconds")

    await simcontrol.create_simstate_condition(cmapi.ConditionSimState.finished).wait()
    await simcontrol.stop_and_disconnect()
    await gpu_sensor.stop()

    cmapi.logger.info(f"Execution with SimControl finished.")

if __name__ == "__main__":
    cmapi.Task.run_main_task(main())
